# v0.2.0 — 인증 시스템 & 공통 레이아웃 (2026-02-21)

## 개요

사용자 로그인/로그아웃 흐름과 인증된 사용자만 접근할 수 있는 보호 라우트,  
그리고 모든 페이지에서 공유되는 Header와 레이아웃을 구축하는 단계.

---

## 인증 전략: 세션 기반

백엔드가 Spring Security의 **세션 쿠키** 방식을 사용하므로, 클라이언트는 JWT를 직접 다루지 않음.

| 항목        | 선택                  | 이유                        |
| ----------- | --------------------- | --------------------------- |
| 인증 방식   | 세션 쿠키             | 백엔드 Spring Security 호환 |
| 토큰 저장소 | 없음 (쿠키 자동 관리) | XSS 공격에 안전             |
| 상태 관리   | Zustand `authStore`   | 로그인 여부를 전역으로 참조 |

**axios 인터셉터 설정**

```js
// api/axiosInstance.js
const axiosInstance = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  withCredentials: true, // ← 이 설정이 핵심
});
```

`withCredentials: true` 없이는 쿠키가 CORS 요청에 포함되지 않음.  
Spring Security의 세션 쿠키 기반 인증이 동작하려면 필수.

## authStore 설계

```js
// store/authStore.js
const useAuthStore = create((set) => ({
  user: null,
  isAuthenticated: false,
  login: (user) => set({ user, isAuthenticated: true }),
  logout: () => set({ user: null, isAuthenticated: false }),
}));
```

**왜 단순하게 유지했나?**

- 서버 세션이 실제 인증 상태를 관리 → 클라이언트 store는 UI 렌더링 목적으로만 사용
- 새로고침 시 `GET /api/users/me`를 호출해서 세션을 복원 → store를 DB처럼 쓰면 안 됨

## ProtectedRoute 패턴

```jsx
// components/guards/ProtectedRoute.jsx
const ProtectedRoute = ({ children }) => {
  const { isAuthenticated } = useAuthStore();
  if (!isAuthenticated) return <Navigate to="/login" replace />;
  return children;
};
```

**`replace` prop을 쓰는 이유**  
`replace` 없이 redirect하면 브라우저 히스토리에 `/` → `/login` 순서로 쌓임.  
로그인 후 Back 버튼을 눌렀을 때 `/login`으로 돌아오는 이상한 UX가 발생.  
`replace`를 쓰면 히스토리에서 덮어쓰기 → 자연스러운 뒤로가기 동작 보장.

## 중첩 라우트 (Nested Routes) 패턴

처음엔 각 보호 페이지마다 `<ProtectedRoute>`로 감쌌으나, 리팩토링하여 중첩 라우트 적용.

```jsx
// 리팩토링 전
<Route path="/" element={<ProtectedRoute><Dashboard /></ProtectedRoute>} />
<Route path="/scores" element={<ProtectedRoute><Scores /></ProtectedRoute>} />

// 리팩토링 후
<Route element={<ProtectedRoute><ProtectedLayout /></ProtectedRoute>}>
  <Route path="/" element={<Dashboard />} />
  <Route path="/scores" element={<Scores />} />
</Route>
```

**왜 중첩 라우트인가?**

- Header를 매 페이지마다 `import`할 필요 없음 → `ProtectedLayout`이 `<Outlet />`으로 자식 렌더링
- 인증 로직을 부모 라우트 한 곳에서만 처리 → DRY(Don't Repeat Yourself)
- 레이아웃 변경(Header 디자인 수정 등)이 `ProtectedLayout` 한 파일만 건드리면 됨

## 공통 컴포넌트 설계 원칙

### Button

```jsx
<Button variant="primary" size="md" loading={isLoading}>
  로그인
</Button>
```

- `variant`: primary / secondary / ghost / danger — 상황별 시각적 구분
- `size`: sm / md / lg — 컨텍스트별 크기 조정
- `loading`: true일 때 Spinner 표시 + 버튼 비활성화 → 중복 제출 방지

**왜 props로 변형을 제어하나?**  
CSS 클래스를 외부에서 직접 넘기면(`className="bg-red-500"`) 컴포넌트 일관성이 깨짐.  
허용된 `variant` 값만 사용하도록 강제하면 디자인 시스템이 유지됨.

### Input

- `label`, `error`, `helperText`를 내장 → 폼마다 label/error 마크업을 반복하지 않음
- `error` prop이 있으면 빨간 테두리 자동 적용

## 세션 복원 흐름

앱이 처음 로드될 때(`App.jsx useEffect`) `GET /api/users/me`를 호출:

```
앱 로드
  └→ GET /api/users/me
       ├─ 200 OK → authStore.login(user) → 마지막 페이지 또는 /
       └─ 401 → authStore 그대로 (isAuthenticated: false)
                → ProtectedRoute가 /login으로 redirect
```

**왜 매 앱 로드마다 API를 호출하나?**  
Zustand store는 새로고침하면 초기화됨.  
세션 쿠키는 남아있는데 store가 비어있으면 무조건 로그인 화면으로 튕김 → UX 불량.  
서버에서 현재 세션 상태를 정답으로 보고 화면 상태를 동기화하는 것이 정석.
