# v0.4.0 — 대시보드 (2026-02-26)

## 개요

사용자가 앱에 진입했을 때 첫 화면(`/`)에서 자신의 플레이 통계를 한눈에 볼 수 있는  
대시보드 페이지를 구현. 백엔드에 별도 통계 API가 없어서 클라이언트에서 조합.

---

## 핵심 문제: 통계 API 없음

백엔드는 `/api/scores` 하나만 존재. `GET /api/dashboard` 같은 통계 전용 API 없음.

**두 가지 선택지:**

1. 백엔드에 통계 API 추가 (서버 수정 필요)
2. 클라이언트에서 여러 번 `/api/scores`를 호출해 조합

→ **2번 선택**: 프론트엔드 학습 목적상, 클라이언트에서 데이터 조합 패턴을 익히는 것이 더 유익.

## Promise.all 병렬 API 호출

```js
// 나쁜 예: 순차 호출 (느림)
const totalRes = await scoresApi.getScores({ size: 1 }); // 500ms 대기
const fcRes = await scoresApi.getScores({ clearType: "FULL_COMBO", size: 1 }); // 또 500ms 대기
// 총 약 3초 소요 (API 6개 × 500ms)

// 좋은 예: 병렬 호출 (빠름)
const [totalRes, fcRes, exHardRes, hardRes, clearRes, recentRes] =
  await Promise.all([
    scoresApi.getScores({ size: 1 }),
    scoresApi.getScores({ clearType: "FULL_COMBO", size: 1 }),
    scoresApi.getScores({ clearType: "EX_HARD_CLEAR", size: 1 }),
    scoresApi.getScores({ clearType: "HARD_CLEAR", size: 1 }),
    scoresApi.getScores({ clearType: "CLEAR", size: 1 }),
    scoresApi.getScores({ page: 0, size: 5 }),
  ]);
// 총 약 500ms 소요 (가장 오래 걸린 API 하나 기준)
```

**왜 `size: 1`인가?**  
통계에 필요한 건 `totalElements`(전체 개수)뿐이므로, 실제 콘텐츠는 1개만 받아도 됨.  
100개씩 받아서 `.length`를 세는 것보다 네트워크 효율이 압도적으로 좋음.

## useDashboard 훅 설계

```js
const useDashboard = () => {
  const { isAuthenticated } = useAuthStore();

  const fetchDashboardData = useCallback(async () => {
    if (!isAuthenticated) {
      // ← 비인증 시 API 호출 방지
      setIsLoading(false);
      return;
    }
    // 병렬 fetch...
  }, [isAuthenticated]);

  useEffect(() => {
    fetchDashboardData();
  }, [fetchDashboardData]);

  return { stats, recentScores, isLoading, error, refetch: fetchDashboardData };
};
```

**`useCallback` + `useEffect` 조합을 쓰는 이유:**

- `fetchDashboardData`를 `useCallback`으로 메모이제이션 → `useEffect`의 의존성 배열에 안전하게 추가
- `useCallback` 없이 함수를 `useEffect` 안에 직접 정의하면 매 렌더마다 새 함수 → 무한 루프 위험
- 또는 `refetch` 기능이 필요해서 함수를 밖으로 꺼낼 때 `useCallback` 필수

**왜 `isAuthenticated`를 조건 체크하나?**  
ProtectedRoute가 있어도 Zustand store가 초기화 중인 짧은 순간에  
인증되지 않은 상태로 훅이 실행될 수 있음. API 호출 낭비 방지.

## 클리어 분포 차트: CSS 전용 구현

별도 차트 라이브러리(Chart.js 등) 없이 CSS Flexbox만으로 구현.

```jsx
<div className="h-6 w-full flex rounded-full overflow-hidden bg-slate-700">
  <div
    style={{ width: `${fcPercent}%` }}
    className="bg-gradient-to-r from-pink-500 to-yellow-400"
  />
  <div style={{ width: `${exHardPercent}%` }} className="bg-yellow-400" />
  <div style={{ width: `${hardPercent}%` }} className="bg-white" />
  <div style={{ width: `${clearPercent}%` }} className="bg-blue-400" />
  <div style={{ width: `${otherPercent}%` }} className="bg-slate-500" />
</div>
```

**왜 차트 라이브러리를 쓰지 않았나?**

- 스택 바(Stack Bar Chart) 하나를 위해 Chart.js(번들 크기 ~200KB)를 설치하는 건 과함
- CSS Flex + percentage width는 동일한 시각 효과를 0KB 추가 비용으로 달성
- `transition-all duration-500`으로 부드러운 로딩 애니메이션 자동 적용

**퍼센트 계산 주의점:**

```js
const otherPercent = Math.max(
  0,
  100 - (fcPercent + exHardPercent + hardPercent + clearPercent),
);
```

`Math.max(0, ...)` 처리를 안 하면 부동소수점 오차로 음수가 나올 수 있음.  
음수 width는 CSS에서 무시되지만 의도치 않은 시각 오류 방지.

## StatCard 컴포넌트

```jsx
const StatCard = ({ icon: Icon, label, value, colorClass }) => (...)
```

**`icon: Icon` 구조 분해 패턴**  
props 이름은 `icon`(소문자)이지만, JSX에서 컴포넌트로 쓰려면 대문자 시작 변수 필요.  
`:Icon`으로 alias 지정 → `<Icon size={24} />`처럼 직접 렌더링 가능.

## 보안 이슈 해결: application.properties git 기록 삭제

이 버전 개발 중 백엔드 `application.properties`에 DB 비밀번호가 평문으로  
git 커밋 기록에 포함된 것을 발견. 즉시 처리:

1. `git-filter-repo`로 41개 커밋 전체 재작성 → 파일 기록 완전 삭제
2. `.gitignore`에 `application.properties` 추가
3. `application-example.properties` (비밀번호 없는 템플릿) 생성 및 커밋
4. `git push --force`로 GitHub origin 덮어쓰기

**교훈**: 민감 정보는 처음부터 `gitignore`에 추가하거나, 환경 변수로 분리해야 함.  
나중에 발견해서 지우는 것보다 처음부터 안 올리는 것이 훨씬 안전하고 쉬움.
