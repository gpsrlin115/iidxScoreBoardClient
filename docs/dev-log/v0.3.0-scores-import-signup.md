# v0.3.0 — 스코어 목록 & CSV 가져오기 & 회원가입 (2026-02-22~25)

## 개요

핵심 데이터 조회 기능(스코어 목록)과 데이터 입력 경로(CSV 업로드, 회원가입)를 구현.  
백엔드 `/api/scores` 페이지네이션 API를 클라이언트에서 어떻게 소비할지 패턴을 확립.

## 스코어 목록 (`/scores`)

### 커스텀 훅 패턴: `useScores`

컴포넌트(Scores.jsx)가 데이터를 직접 fetch하지 않고 `useScores` 훅에 위임.

```
Scores.jsx (UI만 담당)
  └→ useScores() (데이터 페칭 + 상태 관리)
       └→ scoresApi.getScores() (HTTP 호출)
            └→ GET /api/scores?clearType=CLEAR&page=0&size=20
```

**왜 훅으로 분리하나?**

- 컴포넌트는 "어떻게 보이냐"만, 훅은 "어떤 데이터를"만 책임짐 → 단일 책임 원칙
- 로딩/에러/데이터 상태를 훅 내부에서 관리하면, 컴포넌트가 `isLoading ? <Spinner/> : <Table/>` 만 처리하면 됨
- 동일한 데이터가 필요한 다른 컴포넌트에서 그냥 `useScores()` 재사용 가능

### API 파라미터 클리닝

```js
// api/scores.js
const getScores = (params) => {
  // 빈 문자열(""), null, undefined 값은 쿼리스트링에서 제거
  const cleanedParams = Object.fromEntries(
    Object.entries(params).filter(([, v]) => v !== "" && v != null),
  );
  return axiosInstance.get("/api/scores", { params: cleanedParams });
};
```

**왜 클리닝이 필요한가?**  
`clearType=""` 상태로 보내면 백엔드가 `?clearType=`를 빈 값으로 파싱할 수 있음.  
`undefined`/`null` 파라미터는 axios가 `?clearType=undefined`으로 직렬화하기도 함.  
필터 초기화(전체 보기) 상태에서 서버에 불필요한 파라미터가 전달되는 것을 방지.

### IIDX 클리어 타입 배지

| 클리어 타입   | 배지 색상                | 게임 원본 색상  |
| ------------- | ------------------------ | --------------- |
| FULL_COMBO    | 핑크 → 노랑 (그라디언트) | 레인보우 / 핑크 |
| EX_HARD_CLEAR | 노랑                     | 골드            |
| HARD_CLEAR    | 흰색                     | 흰색            |
| CLEAR         | 파랑                     | 밝은 파랑       |
| EASY_CLEAR    | 초록                     | 초록            |
| ASSIST_CLEAR  | 보라                     | 보라            |
| FAILED        | 회색                     | 회색            |

**AC 라벨 변경**: 초기에 `ASSIST_CLEAR → 'ASS'`로 줄였으나 부적절하다고 판단, `'AC'`로 수정.

## CSV 가져오기 (`/import/csv`)

### 업로드 진행률 구현

`axios`의 `onUploadProgress` 콜백으로 진행률 계산:

```js
// api/import.js
const uploadCsv = (file, onProgress) => {
  const formData = new FormData();
  formData.append("file", file);

  return axiosInstance.post("/api/scores/import", formData, {
    headers: { "Content-Type": "multipart/form-data" },
    onUploadProgress: (event) => {
      const percent = Math.round((event.loaded / event.total) * 100);
      onProgress(percent);
    },
  });
};
```

**초기에 겪은 문제**: `Content-Type`을 수동으로 `multipart/form-data`로 설정했더니 오히려 오작동.  
`boundary` 파라미터가 없어서 서버가 multipart 파트를 파싱 못 함.  
해결: `Content-Type` 헤더를 삭제하면 axios가 FormData를 감지하여 boundary 포함 헤더 자동 생성.

### 드래그 앤 드롭 UX

- `dragover` 이벤트에서 `preventDefault()` 필수 → 기본 동작(파일 열기)을 막아야 drop 이벤트 발생
- drag 진입/이탈 시 테두리 색상 변경으로 시각적 피드백 제공
- `input[type=file]`은 시각적으로 숨기고 레이블 클릭으로 trigge → 스타일 자유도 확보

## 회원가입 (`/signup`)

### 단일 onChange 핸들러 패턴

```jsx
const [form, setForm] = useState({
  username: "",
  email: "",
  password: "",
  confirmPassword: "",
});

const handleChange = (e) => {
  const { name, value } = e.target;
  setForm((prev) => ({ ...prev, [name]: value }));
  // 해당 필드 에러 즉시 초기화
  setErrors((prev) => ({ ...prev, [name]: "" }));
};
```

**Computed Property Name** `[name]: value` — 동적 키로 객체 업데이트.  
필드 4개를 함수 1개로 처리하여 코드 중복 제거.

### 클라이언트 유효성 검사 전략

```
제출 버튼 클릭
  └→ validate() 실행 (클라이언트 검사)
       ├─ 에러 있음 → errors 객체 업데이트 → 서버에 요청 안 보냄
       └─ 에러 없음 → API 호출 → 서버 에러는 toast로 출력
```

**클라이언트 검사를 왜 하나?**  
서버 왕복 없이 즉각 피드백 → 낮은 레이턴시, 서버 부하 감소.  
단, 클라이언트 검사는 우회 가능하므로 서버 검사도 항상 필요함.

### errors 객체 패턴

```js
const [errors, setErrors] = useState({});

// 사용처
<Input name="email" error={errors.email} />;
```

`errors.email`이 `undefined`이면 에러 없음, 문자열이면 에러 메시지 표시.  
필드별 에러를 하나의 객체로 관리하면 `errors.email`, `errors.password`처럼 직관적 접근.

## scoresStore vs useScores 분리

|      | `scoresStore` (Zustand)                           | `useScores` (훅)                     |
| ---- | ------------------------------------------------- | ------------------------------------ |
| 책임 | 필터/페이지 상태 전역 유지                        | API 호출 + 로딩/에러 상태            |
| 이유 | 다른 페이지에서 스코어 필터 상태를 참조할 수 있게 | 데이터 페칭 로직을 컴포넌트에서 분리 |

`/scores` 페이지를 떠났다가 돌아와도 필터 상태가 유지되는 UX를 위해 Zustand에 필터 저장.  
단순히 `useState`를 쓰면 페이지 이동 시 필터가 초기화됨.
